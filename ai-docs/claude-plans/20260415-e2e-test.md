# Plan: Real e2e test for youtube-ai-chat chat flow

## Context

The repo has a single placeholder e2e test (`e2e/placeholder.spec.ts`) that only asserts the app title / subtitle are visible. There is no real chat flow coverage because the app needs:

- a running Bodhi server (normally an installed desktop app + browser extension), and
- an API-model configured against OpenAI so the chat can actually respond.

We want a repeatable end-to-end test that:

1. Boots a Bodhi server in-process using `@bodhiapp/app-bindings` (same NAPI-driven pattern `BodhiSearch/bodhi-browser/sdk-test-app` already uses).
2. Configures an OpenAI `gpt-4.1-nano` API model on that server (cheap, fast — suitable for CI).
3. Points the youtube-ai-chat client at that server in **direct mode** (no browser extension), logs in via Keycloak, sends `"what day comes after monday? answer in one word"`, and asserts the assistant reply contains `tuesday` (case-insensitive).
4. Upgrades the chat UI to the per-turn testid convention from `sdk-test-app`/`ChatSection` so messages can be addressed by turn and type.

All decisions in this plan were confirmed with the user via AskUserQuestion.

---

## Decisions (confirmed with user)

| Area | Choice |
|---|---|
| Client → server connection | **Direct mode** via `BodhiProvider` — pass a pre-constructed `UIClient` init'd with `serverUrl` (no extension). |
| Test user credentials | **Add `BODHIAPP_USERNAME` / `BODHIAPP_PASSWORD`** to `e2e/.env.example` and `.env.test`. |
| API-model setup | **Drive `/ui/models/api/new/`** on the Bodhi server via Playwright (reuse the `ApiModelsPage` flow). |
| Message testids | **Adopt per-turn pattern**: `data-testid="chat-message-turn-${n}"`, `data-messagetype`, `data-turn`. |

## Reusable assets (do not rewrite)

Copy/port from `BodhiSearch/bodhi-browser/sdk-test-app/e2e/`:

- `tests/utils/bodhi-server-manager.ts` — `BodhiServerManager` class; start/stop app-bindings server, pick free port, create temp HOME, wait on `/ping`.
- `tests/pages/LoginPage.ts` — Keycloak form login against the Bodhi server.
- `tests/pages/ApiModelsPage.ts` — configure an OpenAI API model via `/ui/models/api/new/` (`api-key-input`, `prefix-input`, `fetch-models-button`, `available-model-<name>`, `create-api-model-button`).
- `tests/global-setup.ts` pattern — start server, log in, configure model, write state file.
- `ChatSection.sendAndWait` selector/wait pattern — the relevant pieces being: `[data-testid="chat-processing"]` going `hidden`, per-turn message elements, `data-messagetype` → extract assistant reply.

These files will be **copied** into this repo (they live outside this repo's node_modules) and lightly trimmed — multi-tenant support, MCPs, and extension-mode helpers are not needed here.

## Implementation steps

### 1. Dependencies

Add to `package.json` devDependencies (matching `sdk-test-app/e2e`):

- `@bodhiapp/app-bindings` (and the matching platform-specific package, e.g. `@bodhiapp/app-bindings-darwin-arm64` — let npm resolve optional deps).
- `dotenv`
- `cross-env` is already present.

Keep `@playwright/test` at the existing version.

### 2. Env plumbing

**`e2e/.env.example`** — add:

```
BODHIAPP_CLIENT_ID=
BODHIAPP_CLIENT_SECRET=
BODHIAPP_USERNAME=
BODHIAPP_PASSWORD=
BODHIAPP_AUTH_URL=https://main-id.getbodhi.app
BODHIAPP_AUTH_REALM=bodhi
OPENAI_API_KEY=
```

**`e2e/.env.test`** — user populates the above. Not committed (already gitignored via `e2e/.env.example` presence).

### 3. Bodhi server harness

Create `e2e/tests/utils/bodhi-server-manager.ts` — port of the sdk-test-app file, trimmed to standalone deployment only. Public surface: `new BodhiServerManager(config)`, `start() → serverUrl`, `stop()`, `getServerUrl()`.

### 4. Page objects

Create `e2e/tests/pages/`:

- `LoginPage.ts` — verbatim port.
- `ApiModelsPage.ts` — verbatim port.
- `ChatPage.ts` — new, thin wrapper over youtube-ai-chat's UI. Methods:
  - `waitReady()` — wait until the client status is `ready` and auth section is `authenticated`.
  - `selectModel(modelId: string)` — set the model in the existing `[data-testid="model-selector"]` dropdown (trigger + click the option for `oai/gpt-4.1-nano`).
  - `send(prompt: string)` — fill `[data-testid="chat-input"]`, click `[data-testid="send-button"]`.
  - `waitForAssistantTurn(turn: number)` — wait until `[data-testid="chat-processing"]` (added in UI changes below) is hidden AND `[data-testid="chat-message-turn-${turn}"][data-messagetype="assistant"]` is attached.
  - `getAssistantText(turn: number): Promise<string>` — return the assistant message content.

### 5. UI changes for testability

**Critical files:**

- `src/components/chat/MessageBubble.tsx`
  - Change `data-testid="message-${role}"` → `data-testid="chat-message-turn-${turn}"`.
  - Add `data-messagetype={role}` (values: `user`, `assistant`, `thinking`, `tool_call`, `tool_result`).
  - Add `data-turn={turn}`.
  - Accept a `turn: number` prop.

- `src/components/chat/ChatMessages.tsx`
  - Derive a stable per-turn index when iterating messages and pass it to `MessageBubble`.
  - Add `data-testid="chat-processing"` with visibility bound to streaming/loading state (hidden once the assistant turn is fully received). Today a `streaming-indicator` testid exists but only renders during streaming — consolidate or keep both; keep `streaming-indicator` untouched and add a separate `chat-processing` marker that covers the full request lifecycle (submitted → response complete).

- `src/components/Layout.tsx` — no change required.

**Direct-mode wiring in `src/App.tsx`:** If `VITE_BODHI_SERVER_URL` is set at build time, construct a `UIClient` via `@bodhiapp/bodhi-js` with `authServerUrl` + `authClientId`, call `client.init({ serverUrl: VITE_BODHI_SERVER_URL, testConnection: true })`, and pass it to `<BodhiProvider client={client} …>`. When unset, keep current behavior (extension mode). Add the env var to `src/env.ts`.

### 6. Global setup

Create `e2e/tests/global-setup.ts`:

1. Load `e2e/.env.test` via `dotenv`.
2. Validate required env vars; fail fast with a clear message listing any missing ones.
3. `new BodhiServerManager({ port: 0, authUrl, authRealm, clientId, clientSecret, appStatus: 'ready', logLevel: 'debug', logToStdout: true, timeout: 30_000 }).start()`.
4. Open a throwaway Chromium context and:
   - `LoginPage.performOAuthLogin('/ui/chat/')` using `BODHIAPP_USERNAME` / `BODHIAPP_PASSWORD`.
   - `ApiModelsPage.configureApiModel(OPENAI_API_KEY, 'oai/', 'gpt-4.1-nano')`.
5. Write `{ bodhiServerUrl }` to `e2e/.test-state.json` for workers to read.
6. Return a teardown that stops the server and deletes the state file.

### 7. Playwright config

`playwright.config.ts`:

- Set `globalSetup: './e2e/tests/global-setup.ts'`.
- Add a second webServer command that starts Vite with `VITE_BODHI_SERVER_URL` injected from the state file (simplest: read the state file in the config via a small helper, or use an env var set by global-setup before Playwright spawns webServer). If that sequencing is awkward, alternative: have global-setup start Vite directly and skip Playwright's webServer — but preferred approach is to keep Vite under Playwright's webServer and pass the URL via a known env var written out by global-setup and `dotenv`-loaded in `vite.config.ts`.
- Keep existing baseURL / timeouts.

### 8. The real test

Rename `e2e/placeholder.spec.ts` → `e2e/chat.spec.ts`:

```ts
test('chat answers "what day comes after monday?" with tuesday', async ({ page }) => {
  const chat = new ChatPage(page);
  await page.goto('/');
  await chat.waitReady();
  await chat.selectModel('oai/gpt-4.1-nano');
  await chat.send('what day comes after monday? answer in one word');
  await chat.waitForAssistantTurn(0);
  const reply = await chat.getAssistantText(0);
  expect(reply.toLowerCase()).toContain('tuesday');
});
```

Delete the old `#root`-visibility assertion entirely; it was incorrect because `Layout` uses `fixed inset-0`.

## Verification

1. `cp e2e/.env.example e2e/.env.test` and fill credentials.
2. `npm install` (pulls in `@bodhiapp/app-bindings` + platform binary).
3. `npm run test:e2e` — expect the Bodhi server to boot, Keycloak login + model config to run once in global-setup, Vite dev server to come up with `VITE_BODHI_SERVER_URL` set, and the chat spec to pass.
4. Inspect the Playwright trace for the chat turn — the assistant message should contain "Tuesday".
5. `npm run ci:test:e2e` — same, with `CI=true`; confirms retries/workers config works headless.
